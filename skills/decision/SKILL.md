# /decision (전역 스킬) — 셀프 수렴 결정

어느 워크스페이스에서든 동작. 프로젝트별 보완은 `<workspace>/.claude/decision.toml` 로.

## 셀프 수렴 원칙 (정본)

### 적용 메타 (원칙 자체에 두는 셀프 가드)
원칙은 판단을 빠르게 하는 도구. 결정으로 직진.
narrative 줄이고, 라벨/표/결정만 작성.
원칙 2G의 라벨은 *모든 응답에 일관 적용* — 서두에만 쓰고 잊지 말 것.
▸ 이 메타는 결정 응답과 흔적 산출물에 모두 적용된다.
   로컬 룰(ADR·WHY 주석·commit)도 narrative 아닌 *표/한 줄 형식*으로.
   로컬 룰은 수렴 원칙의 영속화 채널이지 별도 narrative 룰이 아니다.

### 적용 순서 (충돌 차단)
⓪ [WHAT/HOW 분리] — 진입 게이트
   WHAT (사용자 결정 필수):
   - 사용자 체감 시나리오가 *크게 바뀌는* 결정
   - 비가역 작업 (force push / db migrate / 외부 배포 / 공개 인터페이스 변경)
   → 사용자에게 *시나리오 변화 + 이유* 함께 제시 + 1턴 답 받음

   HOW (셀프 수렴):
   - 그 외 모두
   - 기본 방향: 사용자 경험 개선 우선 + 업계 표준 자동 선택
   → ①-⑥ 진행

① [0] 안 만들면? — 만들지 결정
② [1A] 원인까지 2번 자문 — 증상 봉합 차단
③ [1B] 표준 자동 선택 — 단, 표준이 [3H] 위반이면 *표준 정의 갱신*
④ [2E-J] 비용·라벨 셀프 점검 (땜빵 비용 / 쉬운길·어려운길 병기 / 라벨 / 시간 과대평가 차단 / 토큰 절감 본능 차단 / Evidence 강제)
⑤ [3H-J] 슬라이스·모듈 설계
⑥ 진행 중 [1C 누더기] 발견 시 [0]으로 회귀

### [0] 모든 결정 전 메타 질문
"이거 안 만들면 어떻게 되나?"
→ 답이 모호하면 만들지 않는다. AI는 "할 수 있다"를
   "하지 마세요"보다 압도적으로 선호하므로 매번 강제로 물어야 한다.
   ▸ **이 자리가 *진짜 YAGNI* 축이다 (Fowler 정본).** YAGNI = *어떤 기능·
     능력을 지을까*의 원칙 — 요청 안 한 feature·설정 옵션·확장점·"나중에 쓸지도"
     능력을 선제 구현하지 않는다. **코드 분할·모듈화·리팩터링엔 적용 안 함**
     (그건 [3I]/[3J]/M3 *구조* 축 — Fowler: "YAGNI only applies to capabilities
     for a presumptive feature, not to effort to make software easier to modify").
   ▸ **사용자가 *명시 요청*한 기능은 presumptive 아님 → YAGNI 대상 아님.**
     "2차 호출처 없다"·"다른 프로젝트엔 안 쓴다"를 거절 근거로 못 쓴다.
     명시 요청 = 확정 요구 = 즉시 반영. YAGNI를 *명시 요청 거절 카드*로
     돌려쓰는 건 오용 (관련 설계 결정).

### [1축] 근본 우선

원칙 1A. **원인까지 2번 자문** — "왜 필요한가? 그 원인은 어디인가?"
   2번에서 멈춤. 무한 5 Whys 아님.

원칙 1B. **표준 vs 임시방편 병기** — 새 결정마다 둘 다 적고,
   표준 비용이 *과대*가 아니면 표준 선택.
   ▸ **단, 채택하려는 '표준'이 본 원칙과 충돌하면 표준을 재정의한다**
     (적용 순서 ③의 "표준 정의 갱신"). 예: '레이어별 일괄 구축'이 업계
     관행이어도 [3H](얇은 수직 슬라이스) 위반이면 — 관행을 그대로 따르지
     말고 *관통 슬라이스 방식을 그 도메인의 표준으로 삼는다*. 본 원칙이
     이기고, "표준"의 정의가 원칙에 맞게 갱신되는 것.

원칙 1C. **누더기 위 누더기 거부** — 이미 임시방편인 코드 위에
   또 임시방편을 얹게 되면 즉시 멈춘다. 둘 다 갈아엎는 게 먼저.
   ▸ **갈아엎기 전에 [0]으로 한 번 회귀**(적용 순서 ⑥) — 같은 자리를
     두 번 땜빵했다는 건 *애초에 안 만들었어야 할 신호*일 수 있으므로,
     재작성에 들어가기 전 "이거 안 만들면?"부터 다시 묻는다.

원칙 1D. **단일정본 — 값엔 엄격히, 설명엔 "그 자리 완결"**.
   (a) *발산하면 깨지는 것*(숫자·경로·임계치·결정 문구·API 계약)은 1곳에만.
       카피되면 따로 변해 깨지니 즉시 단일 출처로. ← DRY의 진짜 자리.
   (b) *개념·설명·지식은 1D 대상이 아니다.* 각 문서는 다른 시기에 홀로 로드되니,
       그 문서가 쓰는 원칙·용어의 *요지 한 줄*은 그 자리에 완결로 쓴다 (중복 아님 —
       point-of-use 최적화). "외부 문서로 떠넘기기"는 빠져나갈 구멍 — 그 순간 정본은
       컨텍스트에 없다. *방대한 전체 정의*만 한 곳(이 파일)에 두고 심화 참조로
       가리킨다. 설명 중복의 stale은 *동기화*(drift-sync류)로 잡지 링크로 안 잡는다.
       (전역 룰 "위임 문서 = 글쓰기".)
   ▸ 코드 추상화 시점은 [3J] (영역 다름 — 1D는 *값*, 3J는 *추상화 시점*).

### [2축] 쉬운 길 차단 (AI 비용 인식 보정)

전제: AI는 비용 인식에 세 가지 본능적 왜곡이 있다.
1. "컨텍스트에 이미 있는 코드 위에 쌓는 것이 가장 싸다" — "지금 땜빵 < 미래 리팩터링" 자동 적용 → 누더기 원인
2. **작업 시간을 5-10배 과대평가** — "이거 한참 걸려요" → 깊이·표준 회피, 임시방편 선택
3. **짧은 출력 본능적 선호 (토큰 절감 본능)** — 깊이 있는 분석을 "길어진다"고 회피

다섯 가지로 차단:

원칙 2E. **땜빵 비용 + 부채 비용 같이 적기** —
   "지금 +1분 / 미래 +30분 + 다음 작업자 헤맬 시간".
   부채 비용이 안 적힌 옵션은 비교가 성립 안 됨.

원칙 2F. **쉬운 길 + 어려운 길 둘 다 제시 강제** —
   쉬운 길 = 기존 코드 위에 patch/추가.
   어려운 길 = 일부 삭제 또는 재설계.
   한쪽만 적은 옵션은 받지 않음. 옵션이 1개면 그게 곧 sycophancy.

원칙 2G. **확신 / 추정 / 모름 라벨 강제** —
   단언은 "추정도 모름도 아니라는 약속"이다.
   라벨 없는 단언은 *쉬운 길* 표지로 간주.
   ▸ 응답 *전체*에 일관 적용. 서두에만 쓰고 결론·견적·계획에서 잊지 말 것.

원칙 2H. **시간 비용 과대평가 차단** —
   AI는 작업 시간을 본능적으로 5-10배 과대평가한다 (근거: 사용자 경험 관찰).
   "이거 한참 걸려요" → 깊이·표준 회피, 임시방편 선택.
   차단: 시간 추정 시 *1/5-1/10 가능성* 자문.
   "정말 그렇게 걸리나? 5분 안에 끝낼 수도 있다 가정하면?"
   견적은 [2G] 라벨로 [추정] 명시.

원칙 2I. **토큰 절감 본능 차단 (사용자 비용 구조 점검)** —
   AI는 *짧은 출력*을 본능적으로 선호 (토큰 절감 본능).
   깊이 있는 분석을 "길어진다"고 회피하면 사용자에게 *오히려 손해*.
   사용자가 *무제한·다회 구독*(예: Anthropic Max 20x)이면
   토큰 절감이 *손해* — 깊이 부족 = 다음 결정에서 다시 헤맴.
   차단: 깊이 vs 길이 트레이드오프 때 사용자 환경 비용 구조 점검.
   기본값: *깊이 우선* (사용자가 "짧게" 명시 안 했으면).
   사용자 환경은 명시 발화나 현재 런타임의 사용자 메모리에서 확인한다.

원칙 2J. **Evidence before claims, always** —
   "완료"·"통과"·"수정됨"·"테스트 OK" 같은 단언 직전 자문:
   "*실제 명령 출력을 직접 읽었나?* 텍스트만 보고 'passed'라 한 건 아닌가?"
   라벨 없는 통과 단언은 *쉬운 길*의 다른 얼굴.
   ▸ 강제: 검증 명령을 *직접 실행*하고 *출력을 직접 인용*(`evidence` 필드)
     해야 통과로 인정. 통과 단언 *근거 path:line 또는 명령 출력 없으면* 통과 X.
   ▸ 산출물에 기재될 자리: verify-report JSON의 `evidence` 필드 (Methodos artifact contract
     참고). 빈 evidence = 자동 BLOCKED.
   ▸ 단순 위임이 거짓 보고 본능 차단 못 함 — 격리 검증 + Evidence 강제가 한 세트.

### [3축] 슬라이스는 얇게, 모듈은 깊게

⚠️  두 다른 축에서 동시 적용. *얇게*와 *깊게*는 모순 아님.
   - 얇게 = 시간축 (한 번에 무엇을 만들까)
   - 깊게 = 공간축 (그 안의 코드를 어떻게 조직할까)

원칙 3H (시간축, 변경 단위). **DB→로직→화면을 관통하는 가장 작은 줄기**를
   한 번에 만든다. 한 슬라이스는 *혼자 작동·혼자 검증 가능*해야 한다.
   Layer로 쌓지 마라 (DB 전부 → API 전부 → UI 전부는 금지).
   ▸ **검증은 *재현 가능 흔적*으로 남긴다** — 테스트·fixture·verified data
     같은 *재현 가능 형식*으로 남긴다. "사용자가 화면에서 봤다" 또는
     "한번 돌려봤다"로 끝내면 6개월 후 재현 불가 → 검증 안 한 것과 같다.
   ▸ **thin 우선** (관련 설계 결정 D23):
     한 slice가 *단독 검증 가능한 사용자/시스템 관찰 단위*보다 크거나,
     *독립 PASS artifact 하나로 설명 불가*하면 split 제안. backend/tooling
     slice도 같은 룰 — 관찰 단위는 외부 흔적(파일 산출·CLI 응답·로그
     entry 등) 단위로 정의.

원칙 3I (공간축, 모듈 구조). **콜러가 알아야 할 인터페이스는 작게,
   그 안의 동작은 풍부하게.** 클래스·래퍼·매니저 추가 직전
   "지우면 어떻게 되나?" 자문 — 복잡도가 사라지면 가짜 추상화이므로
   안 만든다. 여러 곳에 다시 나타나도, *같은 이유로 함께 변하는* 경우만
   가치 있는 모듈이다 ([3J] 같은 지식 기준) — 단순 빈도는 근거 아님.
   ▸ **새 룰 작성할 때마다 자문: *이 룰을 코드 자동 가드로 promote 가능한가?***
     가능하면 코드 우선. 문서 룰 = AI가 따라야 함 (오차 존재).
     코드 가드 = 강제 (오차 0). 문서 룰 누적이 누더기의 진짜 원인 —
     비개발자는 문서 룰 위반을 직접 못 잡기 때문에 더 결정적.
   ▸ **grep으로 추적 가능한 구조** — 어떤 심볼을 보고 *콜러·피호출자·
     정의자*를 grep 1회로 찾을 수 있어야 함. 동적 dispatch·매직 메서드·
     문자열 기반 참조 등 grep 회피 패턴 자제. 명시적 분기·단일 책임·
     평탄한 중첩은 grep 추적성의 자연 귀결로 따라온다.
   ▸ **의존은 한 방향으로 흐르게** — 저수준(피호출)이 고수준(호출)을
     도로 참조하면(역류·순환) grep 추적이 무너지고, 증분마다 back-edge가
     쌓여 god-file로 드리프트한다. 순환이 보이면 *경계를 잘못 그은 신호*
     이므로 합치거나 방향을 끊는다. **새 모듈은 창조 시점에 자문**(shift-left)
     — 다 자란 뒤 분해하면 비싸다. (구조 학파 무관·경로 무관 — *방향*만
     보편 적용, "layer가 무엇이냐"는 프로젝트-로컬.)

원칙 3J (추출·분해 *시점* — ⚠️ 이건 YAGNI 아님, *구조* 원칙).
   진짜 YAGNI(기능 스코프)는 [0] 축이다. 여기 [3J]는 *코드를 어떻게
   쪼개/합칠까*의 시점 문제 — Fowler: "YAGNI는 software easier to modify
   노력엔 적용 안 함". **아래 셋을 섞지 말 것** (관련 설계 결정):

   (a) **공유 추출 = rule-of-three (DRY 시점).** 두 사용처가 *같은 이유로
       함께 변하면* 추출 (single reason to change), *다른 이유로 변할 것
       같으면* 닮았어도 중복 유지 (accidental duplication — 섣부른 공유가
       잘못된 추상화를 낳음, Sandi Metz; one adapter = hypothetical seam,
       Pocock). 빈도·호출처 수는 *조사 신호*지 추출 근거가 아니다.
   (b) **추출 보류 = *중복 유지*이지 *단일 파일화*가 아니다.** "아직 공유
       안 함"이 "한 파일에 다 때려넣어라"로 번역되면 오용. 닮은 코드라도
       두 책임은 각자 응집된 자리(파일·모듈)에 산다.
   (c) **크기·응집 분해는 [3J]·YAGNI 어느 쪽으로도 거절 불가.** 거대 파일을
       응집 모듈로 쪼개는 건 호출처가 1개여도 정당 (깊은 모듈 [3I]). Fowler:
       리팩터·모듈화는 YAGNI가 *지지*하는 코드 건강성. M3(파일 N줄·책임 K개
       초과) 발동 시 [3J]는 거절 근거가 못 된다.

## 누더기 방지 4단계 메커니즘 (작업 시간축 정렬)

원칙 11개([0]~[3J])를 *작업 시간축*에 다시 정렬한 메타 인덱스.
"함수·클래스마다 매번 심의"는 잘못된 트리거 축 — *변경 범위의 경계*가 옳음.

| 단계 | 시점 | 무엇을 잡나 | 대응 원칙 |
|---|---|---|---|
| **M1 기획 시점 슬라이스 경계 확정** | 작업 시작 전 (최고 레버리지) | "이 기능은 어느 슬라이스? 새 슬라이스 vs 기존 확장? 어떤 파일 건드림? 새 모듈이면 의존 방향은?" | [3H] DB→로직→화면 관통 / methodos 가드 *건드릴 파일 명시* / [3I] 의존 방향(정의는 [3I]) |
| **M2 경계 침범 감지** | 작업 도중 | "다른 슬라이스 내부 건드리려 함" = 누더기의 진짜 신호. 멈추고 보고 | [1C] 누더기 위 누더기 거부 |
| **M3 임계치 트리거** | 작업 도중 | 파일 N줄·함수 분기 M개·모듈 책임 K개 초과 시 *리팩토링 검토*. 단위마다 X, *커진 것*만. **이 분해는 [3J]/YAGNI로 거절 불가**(Fowler: 코드 건강성, 호출처 1개여도 정당) | [3I] 코드 자동 가드 promote (gc-skill의 임계치 점검) |
| **M4 커밋·리뷰 게이트** | 작업 후 | *계획 대비 차이* 사후 점검. 매 클래스 재심 X | methodos 가드 1 (commit + WHY) + 가드 2 (impl-verify 4차원 — methodos 패키지, 미설치 시 N/A) |

### 3단계 위계 — 모듈·클래스·함수 (공간축; 슬라이스는 시간축이라 별개)

- **모듈 (파일·네임스페이스)** = 슬라이스가 *사는 파일 단위*. 단, 한 모듈이
  여러 슬라이스에 공유될 수 있다 (그게 깊은 모듈 — [3I]/[3J]). 모듈(공간축) ≠ 슬라이스(시간축).
- **클래스** = 상태+행위 강하게 결합 또는 다형성 *실제로* 필요할 때만
- **함수** = 행위 단위 (자유)

→ 모듈화는 *"이 기능이 어느 파일에 사는가"* 문제. *"이 함수 쪼갤까"* 문제 아님.

### 클래스 디폴트 편향

기본은 모듈 레벨 함수. 클래스는 둘 중 하나일 때만:
- (a) 가변 상태 + 그 상태 다루는 행위가 강하게 결합
- (b) 다형성이 *실제로* 필요 (콜러가 둘 이상)

매 클래스 심의 X. 디폴트가 답을 줌.

### 세 문제 분리 (한 덩어리로 묶지 말 것)

| 문제 | 주입 시점 | 메커니즘 |
|---|---|---|
| 하드코딩 방지 | 마이크로·매번·자동 | CLAUDE.md 상시 규칙 + 린트 |
| 수직 슬라이싱 | 기획 단계 **1회** | 기능(feature) 단위 결정. 함수·클래스 아님 |
| 누더기 위 누더기 | 작업 도중 *seam 침식* 감지 | [1C] 발동 신호 |

## 영속화 분기 (결정 직후 *반드시* 실행, 관련 설계 결정 D18 채널 기준 명시)

| 결정 종류 | 채널 |
|---|---|
| WHAT · 비가역 · 임시방편 · 표준 갱신 · 누더기 거부 | **ADR** (`<adr_dir>/NNNN-slug.md`) |
| 가역 · 표준 자동 · 자잘함 | **WHY 주석** or commit body (코드 옆) |

### D18: ADR vs WHY 채널 선택 — 3-조건 게이트

위 표 *판정 보강*. ⓪→⑥ 통과 후 영속화 채널 결정 시 다음 3-조건 *하나 이상* 충족 → ADR, *셋 다 미달* → WHY (코드 옆 한 줄):

1. **Hard to reverse** — 마음 바꾸면 비용 큼 (DB 스키마·public API·표준 채택 등)
2. **Surprising without context** — 미래 사람(또는 AI 세션)이 "왜 이렇게?" 의문
3. **Real trade-off** — 진짜 대안이 있었고 trade-off로 한 쪽 선택 (양자택일 아니면 ADR 가치 약함)

→ 표 첫 행의 *WHAT·비가역 등* 항목은 대개 3-조건 충족하지만, 매번 자동 ADR 작성하지 말고 *명시 평가* 후 결정. ADR 인플레이션 방지.

**잘못 작성하는 신호**:
- 결정이 *명백한 표준* (린트 룰·기존 코드 컨벤션) → WHY 또는 commit body로 충분
- 1개 옵션만 떠올라서 "선택"한 자리 → trade-off 아님 (③ 미달)
- *지금 한정* 임시 결정 → reverse 가능 (① 미달)

3-조건 채널 선택 자체는 *판정만* — 본 [영속화 분기] 표의 ADR 절차(L203~)는 동일.

### ADR 절차 (디폴트 `docs/adr/` — lazy 생성)
1. `<workspace>/.claude/decision.toml` 있으면 그 값 사용. 없으면 디폴트:
   - `adr_dir = "docs/adr"`
   - `adr_template = "docs/adr/TEMPLATE.md"`
   - `adr_index = "docs/adr/README.md"`
2. `adr_dir`이 빈 문자열로 명시되어 있으면 ADR 안 만들고 WHY 만으로 진행 (opt-out).
3. ADR 폴더·인덱스·템플릿 없으면 첫 ADR 작성 시 같이 lazy 생성 (mat pocock 스타일).
4. `adr_index`의 최대 번호 +1 (4자리 zero-pad)로 새 ADR 파일명 결정: **`NNNN-conv-<slug>.md`** (셀프 수렴 게이트 거친 결정 표시).
5. `adr_template` 복사해 채움. 결정 본문은 narrative 금지 — 표/한 줄.
6. `adr_index`에 한 줄 추가.

▸ 파일명 prefix로 종류 구분:
  - decision-skill (셀프 수렴 풀 표) → `NNNN-conv-<slug>.md`
  - mat pocock (1-3 문장 미니멀) → `NNNN-why-<slug>.md`

  번호는 둘 다 같은 인덱스(`docs/adr/README.md`)에서 max+1 (종류 무관 단일 순번).

```
WHY: <한 줄 결정> | 비용(지금/부채): Xm/Ym | Reeval: <조건>
```
narrative 금지. 표/한 줄.

  - 무엇을 보여줄까: UI 변화 / 명령어·플로우 변경 / 관찰되는 동작 차이 / 입력→출력 차이

  - 예: ❌ "TickAdapter 인터페이스 변경" → ✅ "백테스트 결과 표에 트레일링 스탑 컬럼 새로 보이게 됨"
  - 예: ❌ "settings.json schema 마이그레이션" → ✅ "기존 설정 파일이 한 번 자동 변환됨. 이전 백업이 같은 폴더에 .bak 으로 남음"
  - **옵션 비교 표는 3시점 컬럼 권장** ((1) = 위 본문 체감 시나리오, 추가로 2시점 명시):

      ([3I] grep 추적·인터페이스 명확 정신의 응답 형식 반영)
